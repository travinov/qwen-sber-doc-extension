Подготовьте документацию в стиле Сбера для цели `{{args}}`.
Цель может быть:
- Python-файл (`.py`);
- директория с Python-проектом.

Порядок работы обязателен:

1. Убедитесь, что MCP tools доступны. Если `analyze_python_target` не найден:
   - сообщите, что требуется подключение MCP server `gigadoc-mcp`;
   - остановите выполнение и не генерируйте фиктивный результат.
2. Для Python-файла вызовите `analyze_python_target` для `{{args}}`.
3. Для директории вызывайте `analyze_python_target` в context-safe режиме:
   - `max_modules=40`;
   - `include_modules=false`.
   Если модулей существенно больше, документируйте проект итеративно: сначала обзор, затем отдельные ключевые директории/файлы по запросу.
4. Если результат `kind=module`, вызовите `build_sber_doc_outline`.
5. Если результат `kind=project`, вызовите `build_sber_project_outline`.
6. Сформируйте финальную документацию на русском языке в официальном нейтральном тоне.
7. Для `kind=project` дополнительно включите:
   - обзор структуры директории;
   - список ключевых модулей;
   - проектные точки входа и основные сценарии.
8. Проверьте итог через `validate_sber_doc`.

Требования к стилю:

- без разговорных формулировок;
- без смешения русского и английского в описательных абзацах;
- без домыслов о поведении, не подтвержденных кодом;
- для директории отдельно отмечайте, что документ является обзором проекта, а не построчной спецификацией каждого файла.
- не вставляйте в ответ сырой JSON/дамп tool-результатов; публикуйте только сжатые выводы.

Если входной путь не существует или в директории нет Python-файлов, явно сообщите об этом и не создавайте фиктивную документацию.
